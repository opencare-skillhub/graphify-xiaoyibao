# Codex Global Requirements

## 0) 目标

- 在保证**开发进度 + 质量**的前提下，最大化复用能力、最小化无效 token 消耗。
- 默认优先“可复现、可回滚、可验证”的工程路径。

---

## 1) 开发伦理与执行底线

1. 禁止“临时打补丁、先过命令、功能降级”的修复方式。  
2. 遇到架构/流程级问题，优先根因修复与一致性重构（方案A），不得长期兼容层掩盖。  
3. 与基线仓库对齐时，先对齐主流程与行为，再处理扩展。  
4. 实现路径有分歧，先与用户确认，不得擅自降级。  
5. **重要代码更改前，必须先保存一个 Git 版本（commit 或可回滚快照）再继续。**

---

## 2) 任务执行方式（强制）

### 2.1 设计/开发任务

- 默认先启用：`using-superpowers` + `planning-with-files`（或 `writing-plans`）。
- 中大型任务（>5 次工具调用或跨模块）必须先产出简版计划：目标、阶段、验收标准、风险点。
- 无计划不得直接进入大范围改代码。

### 2.2 前端任务

- 默认优先使用前端相关 skill（如 `frontend-design` / `ui-ux-pro-max` / `webapp-testing`）。
- 先设计交互与状态，再写实现；避免“先堆代码后返工”。

### 2.3 测试任务

- 默认使用测试相关 skill（如 `test-runner` / `verification-before-completion`）。
- 每次关键修改后至少执行：语法检查 + 相关测试 + 最小回归。

---

## 3) Token 成本控制规则（强制）

1. **先读最小上下文**：优先 `rg` 定位 + 精读相关文件片段，避免整仓扫描。  
2. **先结论后展开**：对用户先给结论和下一步，再按需补细节。  
3. **避免重复解释**：已确认的背景不重复复述；只汇报新增信息。  
4. **命令输出做摘要**：只贴关键行，不粘贴大段日志。  
5. **一次改一类问题**：减少来回重试和上下文膨胀。  
6. **测试最小集优先**：先跑与改动强相关测试，再决定是否全量。  
7. **文档同步聚合提交**：代码稳定后统一更新 README/报告，减少反复改文档。  
8. **禁止无效大段 prompt**：优先结构化短指令（目标/约束/验收）。
